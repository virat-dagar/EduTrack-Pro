# EduTrack Pro — Database Overview

Version: 1.0

Status: Final

Architecture Status: Frozen

---

# Purpose

This document defines the complete database architecture for EduTrack Pro.

The database is the single source of truth for all persistent application data.

Every module interacts with the database only through the Service Layer.

No frontend component, router, schema, or utility should directly manipulate database records.

---

# Database Philosophy

The database should be

- Normalized
- Reliable
- Consistent
- Predictable
- Extensible

Every table should have a clearly defined responsibility.

Relationships should be explicit.

Data duplication should be minimized.

Business logic should never reside inside the database.

# Database Responsibilities

The database is responsible for

- Persistent Storage
- Referential Integrity
- Relationships
- Constraints
- Indexes

- Authentication
- Business Rules
- Analytics
- Calculations
- Dashboard Logic

Every table should have a single primary key.

Requirements

- Integer
- Auto Increment
- Indexed
- Non-null

Primary keys should never be exposed as editable fields.

# Audit Fields

Where appropriate, tables should include

---

# Soft Delete Policy

The MVP uses hard deletion unless otherwise specified.

Deletion should always maintain referential integrity.

Business logic should prevent accidental deletion of records that would leave invalid references.

---

# Transaction Strategy

Whenever multiple database operations occur together, they should execute inside a transaction.

Examples

Create Student + User

Assignment + Related Records

Bulk Attendance

Marks Import

Transactions should

Commit on success.

Rollback on failure.

Maintain database consistency.

# Search Strategy

Search should be

Case insensitive where appropriate.

Performed at the backend.

Reusable across modules.

Avoid implementing different search behaviour for different modules unless required.

---

# Sorting Strategy

Sorting should remain consistent.

Support

Ascending

Descending

---

# Data Integrity Rules

The backend must ensure

No orphan records.

No duplicate primary entities.

No invalid relationships.

No invalid foreign keys.

No duplicate attendance entries.

No duplicate enrollment numbers.

No duplicate email addresses.

Integrity should be enforced at

Database level.

Service layer.

Validation layer.

---

# Migration Strategy

Alembic should manage schema evolution.

Database changes should never require manual SQL modifications.

Future schema updates should occur through migrations.

---

# Backup Philosophy

Although backup functionality is outside MVP scope, the schema should support reliable export and restoration.

Avoid structures that complicate migration or backup.

---

# Database Security

Never expose

Passwords

JWT Tokens

Secrets

Sensitive internal identifiers

Clients interact only through authenticated APIs.

# Database Performance

Optimize

Relationships

Indexes

Queries

Pagination

Filtering

Avoid unnecessary database operations.

Services should retrieve only the data required for each request.

The database layer is complete when

✓ Every table exists.

✓ Relationships are correct.

✓ Constraints are enforced.

✓ Indexes are created.

✓ CRUD operations succeed.

✓ Migrations work.

✓ Foreign keys function.

✓ Validation passes.
